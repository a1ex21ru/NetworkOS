Лабораторная работа №4 — Защита

Сетевые сокеты на C (TCP, многоклиентский сервер)

1. Общая архитектура

  • Используется TCP (SOCK_STREAM)
  • IPv4 (AF_INET)
  • Модель: один поток на клиента
  • Каждый клиент имеет независимую структуру состояния
  • Синхронизация — mutex + condition variable

2. setsockopt перед bind — зачем нужен?

Использование:

int opt = 1;
setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

Что делает:

Позволяет повторно использовать порт, если сервер был недавно закрыт.

Почему это важно:

После закрытия TCP-соединения порт остаётся в состоянии TIME_WAIT. Без SO_REUSEADDR повторный bind вызовет ошибку: 
Address already in use

Подводные камни:

  • Не отменяет TIME_WAIT, а разрешает повторный bind
  • Не решает конфликт двух одновременно работающих серверов
  • Нужно вызывать ДО bind()

3. INADDR_ANY

Использование:

address.sin_addr.s_addr = INADDR_ANY;

Что означает:

Сервер принимает соединения на всех сетевых интерфейсах.

Эквивалент:

0.0.0.0

Если указать конкретный IP:

Сервер будет слушать только на этом интерфейсе.

Когда используют:

  • Сервер должен принимать подключения из любой сети
  • Работа в VirtualBox / NAT

4. htons — зачем нужен?

Использование:

address.sin_port = htons(5050);

Что делает:

Host TO Network Short — перевод 16-битного числа из порядка байтов хоста в сетевой (Big Endian).

Почему это нужно:

Разные архитектуры используют разный порядок байтов (Little/Big Endian). Сетевой протокол всегда Big Endian.

Аналогичные функции:

  • htonl — 32-бит
  • ntohs — обратно
  • ntohl — обратно

5. Использованные функции — полный разбор

socket()

Создаёт сокет. Возвращает файловый дескриптор. Подводный камень: возвращает -1 при ошибке.

bind()

Привязывает сокет к IP и порту. Без bind сервер не знает, где слушать.

listen()

Переводит сокет в режим ожидания. Аргумент backlog — размер очереди подключений. Это НЕ количество клиентов.

accept()

Принимает соединение. Возвращает НОВЫЙ сокет для клиента. server_fd остаётся слушающим. Подводный камень: accept блокируется, если нет подключений.

connect()

Устанавливает соединение с сервером (клиент).

send()

Отправляет данные. Не гарантирует отправку всех байтов за один вызов.

recv()

Получает данные. Возвращает:
  • >0 — количество байтов
  • 0 — клиент закрыл соединение
  • -1 — ошибка

close()

Закрывает сокет. После закрытия соединение переходит в TIME_WAIT.

pthread_create()

Создаёт поток.

pthread_mutex_lock / unlock

Защита критической секции.

pthread_cond_wait()

Освобождает mutex и блокируется. После пробуждения снова захватывает mutex. Важно: Используется в цикле while, а не if.

pthread_cond_broadcast()

Будит все ожидающие потоки. Нужно, чтобы каждый поток повторно проверил условие.

6. Возможные вопросы преподавателя

Почему нужен setsockopt?

Чтобы избежать ошибки "Address already in use" после перезапуска сервера.

Что такое INADDR_ANY?

Это 0.0.0.0 — прослушивание всех интерфейсов.

Что будет, если не вызвать htons?

Порт будет интерпретирован неверно из-за порядка байтов.

Что возвращает recv при отключении клиента?

0.

Почему используется TCP, а не UDP?

Нужна надёжная передача и сохранение порядка данных.

Почему используется while в pthread_cond_wait?

Возможны ложные пробуждения, условие нужно проверять повторно.

Есть ли взаимная блокировка?

Нет, используется один mutex без вложенных блокировок.

7. Подводные камни реализации

  • Отсутствие SO_REUSEADDR — ошибка при перезапуске
  • Блокирующий accept может остановить сервер
  • recv блокируется без данных
  • Необработанный SIGPIPE при send
  • Нужно проверять возвращаемые значения функций

8. Улучшения архитектуры

  • Использование select / poll / epoll
  • Thread pool вместо потока на клиента
