## 1. Задание

Некий университет решил продемонстрировать свою политкорректность, применив известную доктрину верховного суда США «Равенство порознь равенством не является» не только к цвету кожи, но и к полу. Результатом этого решения явились совместные ванные комнаты в общежитиях. Тем не менее в поддержку исторически сложившейся традиции университет постановляет, что если в ванной комнате есть женщина, то другая женщина может туда зайти, а мужчина не может, и наоборот. На двери ванной есть индикатор, показывающий, в каком из трёх состояний находится ванная:

- никого нет;
- в ванной женщины;
- в ванной мужчины.

Однако число кабинок в ванной ограничено. Поэтому, если все места в ванной заняты, то туда не может войти ни мужчина, ни женщина. Таким образом, должно также отображаться число свободных мест в ванной комнате. Ваша задача — написать программу, в которой каждый клиент (будь то мужчина или женщина) представляет собой **отдельный процесс** (в данной работе — многопроцессная реализация), а ванная комната — разделяемый ресурс. Организовывать очередь для процессов в программе не требуется: то, в какой последовательности процессы будут пытаться завладеть ресурсом, будет зависеть от планировщика ОС. Запустите программу с разным числом проживающих студентов и разной вместимостью ванны и составьте своего рода график зависимостей (загруженности ванной комнаты от этих параметров, средним временем простаивания в очереди и т. д.). Сколько времени каждый студент проводит в ванной — случайная величина, колеблющаяся в определённых пределах.

## 2. Структуры данных

### 2.1. Перечисление состояний

```c
typedef enum { EMPTY, MALE, FEMALE } BathroomStatus;
```

**Назначение:** определяет текущее состояние ванной комнаты (пусто, занято мужчинами или женщинами) и пол студента.

### 2.2. Структура Bathroom (в разделяемой памяти)

```c
typedef struct {
    pthread_mutex_t access_lock;           // Мьютекс для защиты критической секции (PTHREAD_PROCESS_SHARED)
    pthread_cond_t  availability_signal;   // Условная переменная для ожидания входа (PTHREAD_PROCESS_SHARED)
    unsigned int total_cabins;             // Общее количество кабинок
    unsigned int occupied_cabins;          // Текущее количество занятых кабинок
    BathroomStatus current_status;         // Текущий пол occupants (или EMPTY)
    unsigned int consecutive_entries;       // Счётчик последовательных входов одного пола
    unsigned int max_consecutive_entries;  // Лимит последовательных входов
    bool forced_switch_pending;            // Флаг ожидания смены пола
    BathroomStatus last_gender;            // Пол последней группы, использовавшей ванную
    volatile bool bathroom_open;           // Флаг работы симуляции (для корректного завершения)
    bool header_printed;                   // Флаг однократного вывода заголовка (защита от дублирования при fork)
    time_t start_time;                     // Время начала симуляции
    unsigned int entered_count;            // Количество успешно вошедших
    unsigned int male_entered;              // Вошло мужчин
    unsigned int female_entered;            // Вошло женщин
    double total_wait_time;                // Суммарное время ожидания (для среднего)
} Bathroom;
```

**Назначение:** хранит состояние разделяемого ресурса (ванной) и примитивы синхронизации. Вся структура размещается в **разделяемой памяти** (shm_open + mmap с MAP_SHARED), чтобы к ней имели доступ родительский и все дочерние процессы. Мьютекс и условная переменная инициализируются с атрибутом **PTHREAD_PROCESS_SHARED**, чтобы корректно работать между процессами.

**Отличие от лабораторной №1:** структура **Student** отсутствует — каждый студент является отдельным процессом; его параметры (id, пол, длительность душа) передаются в функцию процесса как аргументы и хранятся в локальных переменных дочернего процесса. Статистика (entered_count, total_wait_time и т. д.) накапливается в самой структуре Bathroom в разделяемой памяти.

## 3. Алгоритм работы

### 3.1. Общая схема

1. **Родительский процесс** создаёт объект разделяемой памяти (shm_open, ftruncate, mmap), инициализирует в ней структуру Bathroom (openBathroom с мьютексом и cond с PTHREAD_PROCESS_SHARED), выводит заголовок один раз (под мьютексом, по флагу header_printed).
2. В цикле для каждого студента родитель вызывает **fork()**. В дочернем процессе вызывается **studentRoutine**(i, gender, shower_duration); дочерний процесс завершается **exit(0)**.
3. Родитель сохраняет PID каждого потомка в массив и затем в цикле вызывает **waitpid(pids[i], NULL, 0)**, ожидая завершения всех дочерних процессов.
4. После завершения всех детей родитель считывает из разделяемой структуры статистику (entered_count, male_entered, female_entered, total_wait_time), выводит результаты, уничтожает мьютекс и cond, выполняет **munmap** и **shm_unlink**.

### 3.2. Действия процесса-студента (studentRoutine)

1. Вызов **enterBathroom**(shared_bathroom, student_id, gender, &wait_time): попытка войти в ванную (при необходимости — ожидание на условной переменной).
2. При успешном входе — имитация душа: **usleep**(shower_duration в микросекундах).
3. **exitBathroom**(shared_bathroom, student_id, gender).
4. **exit(0)** — завершение дочернего процесса.

### 3.3. Проверка условий входа — isEntryAllowed()

- **Занятость:** если `occupied_cabins >= total_cabins`, вход запрещён.
- **Пустая ванная:** если `current_status == EMPTY`, вход запрещён только при активном `forced_switch_pending` и совпадении пола студента с `last_gender` (чтобы пропустить противоположный пол); иначе вход разрешён.
- **Занятая ванная:** вход разрешён только при совпадении пола с `current_status` и отсутствии `forced_switch_pending`.

Проверка выполняется под мьютексом `access_lock`.

### 3.4. Вход в ванную — enterBathroom()

1. Фиксация времени начала ожидания (time).
2. Захват мьютекса **pthread_mutex_lock**(&b->access_lock).
3. Проверка флага **bathroom_open**; при false — разблокировка и возврат.
4. Цикл **while (!isEntryAllowed(b, gender) && b->bathroom_open)** с вызовом **pthread_cond_wait**(&b->availability_signal, &b->access_lock) — атомарное освобождение мьютекса и ожидание; после пробуждения мьютекс снова захватывается.
5. После выхода из цикла — повторная проверка bathroom_open; при false — broadcast, разблокировка, возврат.
6. Вычисление времени ожидания (difftime), обновление состояния ванной (при пустой — сброс серии, установка current_status; инкремент consecutive_entries, при достижении лимита — forced_switch_pending = true), инкремент occupied_cabins и счётчиков статистики.
7. Вывод сообщения о входе, **printDoorStatus**, **pthread_cond_broadcast**, **pthread_mutex_unlock**.

### 3.5. Выход из ванной — exitBathroom()

1. Захват мьютекса.
2. Декремент **occupied_cabins**.
3. Если **occupied_cabins == 0**: запись **last_gender = current_status**, **current_status = EMPTY**, **forced_switch_pending = false** (предотвращение дедлока при смене пола).
4. Вывод сообщения о выходе, printDoorStatus.
5. **pthread_cond_broadcast**, **pthread_mutex_unlock**.

Использование **broadcast** гарантирует, что все ожидающие процессы перепроверят условия входа после любого изменения состояния.

## 4. Механизм предотвращения голодания

Реализован тот же двухуровневый механизм, что и в лабораторной №1:

- **Уровень 1 — forced_switch_pending:** после достижения лимита max_consecutive_entries входов одного пола устанавливается флаг forced_switch_pending; он блокирует дальнейший вход этого пола до полного освобождения ванной (occupied_cabins == 0). При освобождении ванной запоминается last_gender.
- **Уровень 2 — last_gender при пустой ванной:** при попытке войти в пустую ванную при активном forced_switch_pending вход запрещён для того же пола, что и last_gender; таким образом приоритет отдаётся противоположному полу.

При полном освобождении ванной в exitBathroom флаг forced_switch_pending сбрасывается, чтобы избежать дедлока (ожидающие противоположного пола смогут войти).

## 5. Функции для работы с процессами и разделяемой памятью

### 5.1. Создание процессов — fork()

**Прототип:** `pid_t fork(void);`

**Назначение:** создаёт дочерний процесс — копию родительского. Дочерний процесс получает копию адресного пространства; разделяемая память, отображённая с флагом **MAP_SHARED**, остаётся общей: изменения видны всем процессам, отобразившим эту область.

**Возвращаемое значение:** в родителе — PID дочернего процесса (> 0); в дочернем — 0; при ошибке — -1, errno устанавливается.

**Применение в программе:** для каждого студента вызывается fork(). В родителе PID сохраняется в pids[i]. В дочернем процессе (pid == 0) вызывается studentRoutine(i, gender, shower_duration), затем exit(0).

### 5.2. Ожидание завершения процесса — waitpid()

**Прототип:** `pid_t waitpid(pid_t pid, int *status, int options);`

**Назначение:** приостанавливает выполнение текущего процесса до изменения состояния дочернего процесса (завершение и т. п.).

**Параметры:** pid — идентификатор ожидаемого процесса; status — указатель на код завершения (в программе передаётся NULL); options — 0 (блокирующее ожидание).

**Применение:** после создания всех потомков родитель выполняет цикл waitpid(pids[i], NULL, 0), гарантируя завершение всех студентов до вывода итогов и освобождения разделяемой памяти.

### 5.3. Завершение процесса — exit()

**Прототип:** `void exit(int status);`

**Назначение:** немедленно завершает вызывающий процесс. Код status может быть получен родителем через waitpid. Разделяемая память остаётся доступной другим процессам до munmap/shm_unlink.

**Применение:** в дочернем процессе после выполнения studentRoutine() вызывается exit(0).

### 5.4. Создание/открытие разделяемой памяти — shm_open()

**Прототип:** `int shm_open(const char *name, int oflag, mode_t mode);`

**Назначение:** создаёт или открывает объект разделяемой памяти POSIX по имени.

**Параметры:** name — имя (в программе "/bathroom_shm"); oflag — O_RDWR | O_CREAT; mode — права доступа (0666).

**Применение:** перед mmap вызываются shm_unlink(SHM_NAME) (удаление старого объекта при повторном запуске), затем shm_open; полученный дескриптор используется в ftruncate и mmap.

### 5.5. Задание размера объекта — ftruncate()

**Прототип:** `int ftruncate(int fd, off_t length);`

**Назначение:** задаёт размер объекта разделяемой памяти. В программе: ftruncate(fd, sizeof(Bathroom)).

### 5.6. Отображение в адресное пространство — mmap()

**Прототип:** `void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);`

**Назначение:** отображает объект разделяемой памяти (или файл) в адресное пространство процесса.

**Параметры:** addr — NULL (ядро выбирает адрес); length — sizeof(Bathroom); prot — PROT_READ | PROT_WRITE; flags — **MAP_SHARED** (изменения видны всем процессам, отобразившим эту область); fd — дескриптор от shm_open; offset — 0.

**Применение:** родитель выполняет mmap; указатель сохраняется в shared_bathroom. После fork() дочерние процессы наследуют отображение и видят ту же область, поэтому все работают с одной структурой Bathroom.

### 5.7. Удаление отображения — munmap()

**Прототип:** `int munmap(void *addr, size_t length);`

**Назначение:** удаляет отображение области памяти. После вызова обращаться к ней нельзя.

**Применение:** в родительском процессе после вывода результатов и уничтожения мьютекса/cond вызывается munmap(shared_bathroom, sizeof(Bathroom)).

### 5.8. Удаление объекта по имени — shm_unlink()

**Прототип:** `int shm_unlink(const char *name);`

**Назначение:** удаляет именованный объект разделяемой памяти. Вызывается в конце (после munmap) для освобождения ресурсов ядра.

### 5.9. Инициализация мьютекса и условной переменной для межпроцессного использования

Мьютекс и условная переменная должны располагаться в разделяемой памяти и создаваться с атрибутом **PTHREAD_PROCESS_SHARED**:

- **pthread_mutexattr_init** / **pthread_mutexattr_setpshared**(attr, PTHREAD_PROCESS_SHARED) / **pthread_mutex_init**(&b->access_lock, &attr);
- **pthread_condattr_init** / **pthread_condattr_setpshared**(attr, PTHREAD_PROCESS_SHARED) / **pthread_cond_init**(&b->availability_signal, &attr);

В конце работы вызываются **pthread_mutex_destroy** и **pthread_cond_destroy** перед munmap.

## 7. Вывод

В ходе работы реализована **многопроцессная** программа, моделирующая доступ к общей ванной комнате. В отличие от лабораторной №1 (многопоточная реализация), каждый студент представлен **отдельным процессом**, создаваемым через **fork()**. Обмен данными между процессами выполняется через **разделяемую память** POSIX: **shm_open()**, **ftruncate()**, **mmap()** с флагом **MAP_SHARED**. Синхронизация доступа к разделяемой структуре Bathroom обеспечивается мьютексом и условной переменной, размещёнными в этой же области и инициализированными с атрибутом **PTHREAD_PROCESS_SHARED**. Освоены создание и завершение процессов (fork, exit), ожидание потомков (waitpid), создание и освобождение разделяемой памяти (shm_open, mmap, munmap, shm_unlink), а также межпроцессная синхронизация средствами POSIX (мьютекс и условная переменная с PTHREAD_PROCESS_SHARED). Логика задачи (индикатор на двери, ограничение по полу и по числу кабинок, механизм предотвращения голодания) сохранена и согласована с первой лабораторной работой.
